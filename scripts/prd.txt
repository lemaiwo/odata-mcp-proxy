# Overview

This project creates an MCP (Model Context Protocol) server that exposes **all** SAP BTP Integration Suite administration OData APIs as MCP tools. Unlike the existing `cpi-mcp-server` by vadimklimov which only covers integration content read operations, this server provides full CRUD access to all six OData API categories available at https://api.sap.com/package/CloudIntegrationAPI/odata — enabling AI-powered administration of SAP BTP Integration Suite.

The target user is an **SAP Integration Suite administrator** who wants to use AI assistants (Claude, etc.) to manage, monitor, and troubleshoot their Cloud Integration tenant. This is NOT for business end users consuming integration flows, but for the people maintaining them.

The project reuses the proven architecture from `btp-sap-odata-to-mcp-server` (by lemaiwo): a Node.js/TypeScript MCP server that runs on SAP BTP Cloud Foundry, uses BTP Destinations for connectivity and authentication, and registers tools via a static tool registry. However, instead of dynamically discovering OData services from an SAP backend, this server uses a **hardcoded static list** of all Cloud Integration OData API entity sets with their known schemas.

# Core Features

## 1. Static Tool Registry for Cloud Integration APIs
- All six Cloud Integration OData API categories are registered as MCP tools at startup
- No dynamic discovery needed — the API surface is well-known and stable
- Each entity set gets appropriate tools (list, get, create, update, delete) based on what the API supports
- Tool schemas are defined statically in code, matching the OData entity metadata from the SAP API Business Hub

## 2. Full Cloud Integration OData API Coverage
The following six API categories and their entity sets must be supported:

### Integration Content API (`/api/v1/`)
- **IntegrationPackages** — CRUD on integration packages
- **IntegrationDesigntimeArtifacts** — CRUD on design-time artifacts (iflows, value mappings, etc.) with sub-resources: Resources, Configurations
- **IntegrationRuntimeArtifacts** — Read deployed artifacts and their status, deploy/undeploy operations
- **ValueMappingDesigntimeArtifacts** — CRUD on value mapping design-time artifacts with ValMapSchema navigation
- **MessageMappingDesigntimeArtifacts** — CRUD on message mapping artifacts
- **ScriptCollectionDesigntimeArtifacts** — CRUD on script collection artifacts
- **CustomTagConfigurations** — Read/write custom tag configurations
- **BuildAndDeployStatus** — Read build and deploy status

### Message Processing Logs API (`/api/v1/`)
- **MessageProcessingLogs** — Read message processing logs with navigation to: MessageProcessingLogAttachments, ErrorInformations, AdapterAttributes, MessageStoreEntries, CustomHeaderProperties
- **IdMapFromId2s** — Read ID mapping entries
- **IdempotentRepositoryEntries** — Read idempotent repository entries

### Message Stores API (`/api/v1/`)
- **MessageStoreEntries** — Read entries from the message store
- **NumberRanges** — Read/write number range configurations
- **DataStoreEntries** — CRUD on data store entries
- **Variables** — CRUD on variables
- **JmsBrokers** — Read JMS broker information
- **JmsResources** — Read JMS resource information (queues, etc.)

### Log Files API (`/api/v1/`)
- **LogFiles** — Read HTTP and trace log files
- **LogFileArchives** — Read archived log files

### Partner Directory API (`/api/v1/`)
- **Partners** — CRUD on partner entries
- **StringParameters** — CRUD on string parameters
- **BinaryParameters** — CRUD on binary parameters
- **AlternativePartners** — CRUD on alternative partner entries
- **AuthorizedUsers** — CRUD on authorized user entries

### Security Content API (`/api/v1/`)
- **KeystoreEntries** — Read/manage keystore entries (certificates, key pairs)
- **CertificateResources** — Read certificate chain details
- **SSHKeyResources** — Read SSH key details
- **UserCredentials** — CRUD on user credential artifacts
- **OAuth2ClientCredentials** — CRUD on OAuth2 client credential artifacts
- **SecureParameters** — CRUD on secure parameter artifacts
- **CertificateUserMappings** — CRUD on certificate-to-user mappings
- **AccessPolicies** — CRUD on access policies with ArtifactReferences navigation

## 3. BTP Destination-Based Connectivity
- Hostname and authentication credentials come from a BTP Destination (same pattern as btp-sap-odata-to-mcp-server)
- Supports OAuth2 Client Credentials grant type via a Process Integration Runtime service instance
- Single destination for all API calls (all APIs share the same base URL and credentials)
- Local development support via environment variables or default-env.json

## 4. BTP Cloud Foundry Deployment
- Packaged as an MTA (Multi-Target Application) for deployment to BTP Cloud Foundry
- Uses XSUAA for user authentication
- Health check endpoint
- Configurable logging

## 5. MCP Protocol Compliance
- Implements MCP protocol using the official `@modelcontextprotocol/sdk`
- Supports Streamable HTTP transport (for BTP deployment)
- Supports stdio transport (for local Claude Desktop usage)
- Session management for multi-turn conversations

# User Experience

## User Persona: SAP Integration Suite Administrator
- Manages integration packages, flows, value mappings, and scripts
- Monitors message processing and troubleshoots errors
- Manages security artifacts (certificates, credentials, keystores)
- Configures partner directory entries for B2B scenarios
- Reviews log files for debugging

## Key User Flows

### Monitoring & Troubleshooting
1. "Show me all failed messages from today" → queries MessageProcessingLogs with $filter
2. "What's the error for message ID X?" → navigates to ErrorInformations
3. "Show me the latest log files" → queries LogFiles
4. "Which artifacts are in error state?" → queries IntegrationRuntimeArtifacts with status filter

### Content Management
1. "List all integration packages" → queries IntegrationPackages
2. "Show me the configuration of iflow X" → queries Configurations navigation property
3. "Deploy this integration flow" → POST to deploy endpoint
4. "Update the externalized parameter X to value Y" → PATCH on Configurations

### Security Administration
1. "Show me all expiring certificates" → queries KeystoreEntries with validity filter
2. "List all user credential artifacts" → queries UserCredentials
3. "Create a new OAuth2 client credential" → POST to OAuth2ClientCredentials
4. "Which access policies exist?" → queries AccessPolicies

### Partner Directory Management
1. "List all partners" → queries Partners
2. "Add a string parameter for partner X" → POST to StringParameters
3. "Update the authorized users for partner X" → manages AuthorizedUsers

## Technology Stack
- **Runtime**: Node.js 18+ with TypeScript 5.0+
- **MCP SDK**: `@modelcontextprotocol/sdk` (latest)
- **HTTP Framework**: Express.js
- **SAP Cloud SDK**: `@sap-cloud-sdk/connectivity`, `@sap-cloud-sdk/http-client`
- **SAP Auth**: `@sap/xsenv`, `@sap/xssec`
- **Validation**: Zod
- **Logging**: Winston + `@sap/logging`
- **Build**: TypeScript compiler, MTA Build Tool (MBT)
- **Deployment**: SAP BTP Cloud Foundry via MTA

## Data Model
All entity types and their properties are defined statically in TypeScript interfaces, matching the OData metadata from the SAP API Business Hub. The OData APIs are OData V2 using JSON format.

### Key OData Conventions
- Base path: `https://<tmn-host>/api/v1/`
- CSRF protection: Fetch token via `GET` with `X-CSRF-Token: Fetch` header, send back on mutating requests
- Pagination: `$top`, `$skip`, `$inlinecount`
- Filtering: `$filter` with OData V2 filter syntax
- Selection: `$select` for field projection
- Expansion: `$expand` for navigation properties
- Ordering: `$orderby`

## APIs and Integrations
- **SAP BTP Destination Service**: Resolves destination configuration (URL, auth)
- **SAP Process Integration Runtime**: Provides OAuth2 Client Credentials for Cloud Integration API access
- **XSUAA**: Authenticates MCP server users (administrators)

## Infrastructure Requirements
- SAP BTP subaccount with Cloud Foundry enabled
- SAP Integration Suite subscription with Cloud Integration capability
- Process Integration Runtime service instance (plan: `api`) with roles:
  - `WorkspacePackagesRead` / `WorkspacePackagesEdit` (for integration content)
  - `MonitoringDataRead` (for monitoring/logs)
  - `WorkspaceArtifactsDeploy` (for deploy operations)
  - `AuthGroup_Administrator` or specific security roles (for security content)
- Destination service instance (lite plan)
- Connectivity service instance (lite plan, if cloud connector needed)
- XSUAA service instance (application plan)

# Development Roadmap

Set up the project scaffolding, BTP deployment configuration, and core connectivity layer.

- Initialize Node.js/TypeScript project with MCP SDK, Express, SAP Cloud SDK dependencies
- Create the MTA deployment descriptor (`mta.yaml`) with required BTP service bindings (destination, xsuaa)
- Create `xs-security.json` with appropriate scopes and role templates
- Implement the destination service module that resolves the BTP destination and obtains OAuth2 tokens
- Implement the base SAP HTTP client that:
  - Fetches and caches CSRF tokens
  - Sends authenticated OData V2 requests (GET, POST, PATCH, DELETE)
  - Handles OData error responses and extracts meaningful error messages
  - Supports OData query options ($filter, $select, $expand, $top, $skip, $orderby, $inlinecount)
- Implement configuration management (environment variables, logging)
- Implement health check endpoint
- Create `.env.example` and `example-default-env.json` for local development

- Build the static tool registry framework:
  - Define a tool definition format (entity name, supported operations, input schema, description)

# Logical Dependency Chain

1. **Phase 1 (Foundation)** must come first — everything depends on the HTTP client, destination service, and project setup
2. **Phase 2 (MCP Server & Registry)** depends on Phase 1 — the tool registry framework needs the SAP client to execute calls
3. **Phases 3, 4, 5 (API Tools)** can be developed in parallel once Phase 2 is complete — they are independent API categories each using the same registry framework. However, **Phase 3 (Integration Content)** is recommended first as it is the most commonly used API and provides the quickest visible value
4. **Phase 6 (Polish)** depends on all previous phases being functionally complete

The fastest path to something usable:
- Phase 1 → Phase 2 → Phase 3 gives a working MCP server with integration content management (the most-used admin capability)
- Phases 4 and 5 can then be added incrementally

# Risks and Mitigations

## CSRF Token Handling
- **Risk**: Cloud Integration APIs require CSRF tokens for all mutating operations (POST, PUT, PATCH, DELETE). Token caching and refresh logic can be tricky.
- **Mitigation**: Implement a CSRF token manager that fetches tokens before write operations and retries on 403 (token expired).

## OData V2 Complexity
- **Risk**: OData V2 has complex filter syntax, navigation properties, and batch operations that differ from REST conventions.
- **Mitigation**: Build a robust OData query builder. Start with simple $filter/$select support and add complexity incrementally.

## Tool Count vs. LLM Context
- **Risk**: With ~35+ entity sets each having up to 5 operations, the total tool count could reach 100+, which may overwhelm LLM context windows.
- **Mitigation**: Group related operations (e.g., one tool per entity with an `operation` parameter) or implement a hierarchical discovery pattern similar to btp-sap-odata-to-mcp-server. Consider making tool registration configurable so admins can enable only the API categories they need.

## API Authorization Roles
- **Risk**: Different API categories require different authorization roles on the Process Integration Runtime service instance. Missing roles cause 403 errors.
- **Mitigation**: Document required roles clearly per API category. Implement graceful error handling that tells the user which role is missing.

## Binary Content Handling
- **Risk**: Some operations involve binary content (downloading/uploading artifact resources, certificates, log files). MCP tools return text content.
- **Mitigation**: For downloads, return base64-encoded content or metadata with size info. For uploads, accept base64-encoded content as input.

## MVP Scope
- **Risk**: Building all 6 API categories with full CRUD is a large scope.
- **Mitigation**: MVP should cover Phase 1-3 (foundation + integration content). This covers the most common admin workflows and validates the architecture. Phases 4-5 add monitoring and security capabilities.
